## 一、合夥人 Pre-mortem 共識

### F1 — 資料淺薄死 — 三年後我們累積的資料一文不值

- **情境**：2029 年。某 AI 大廠推出 vertical locksmith agent，用公開知識 + 更強模型，準確率比我們高。我們三年累積的 30 萬筆 ProblemCard 沒結構化成 ML signal，只是 raw log。投資人問：你的 data moat 在哪？我們答不出來。
- **死因關鍵**：資料只當 audit 用，沒當 training & evaluation signal
- **防禦武器**：§B1 資料主權層 / §D1 資料護城河 / §E1 Bronze→Silver→Gold Medallion 三層資料管線
- **現況**：紅燈 — SOW 寫了 Medallion 但實際只到 Bronze；ETL 升 Silver/Gold 未閉環
- **合夥人投票**：× 認為過慮
- **會議備註**：
  > 1. 增加未來副廠或是當地維修廠商可以累積的參考依據
  > 2. phase0 

### F2 — 知識被技術綁架死 — 換 LLM 廠商等於過去三年 SOP 全重寫

- **情境**：當前主力 LLM 突然停服或漲價 10 倍。我們發現 SKILL.md 內混了 vendor-specific 格式、prompt template 跟特定 function-calling spec 綁死。換廠商要重寫 1500 條 SKILL.md，工程估時 6 個月。期間 AI 客服降級到人工。
- **死因關鍵**：知識存在「執行框架」而非「資料層」
- **防禦武器**：§B3 政策與契約層（vendor-neutral markdown）/ §D2 知識護城河 / §E2 7 層記憶（Episodic / Semantic 永恆）
- **現況**：黃燈 — SKILL.md 是 markdown 但 Eval set / Tool 介面綁定特定廠商；可攜性未驗證過
- **合夥人投票**：× 認為過慮
- **會議備註**：

  > llm 廠商可以被抽換
  > skill 和 llm 需要解偶

### F3 — HITL 邊界漂移死 — AI 自作主張承諾退款，一次客訴搞垮品牌

- **情境**：客戶報修後，AI 為了讓客戶開心，承諾「這次免費保固」。實際上保固已過期。客戶拿截圖去消保官 / 媒體。一週後品牌信任崩盤，新客上門率 -40%。我們花一年蓋的客服系統一夜歸零。
- **死因關鍵**：AI 不可決策清單寫在 prompt 而非合約契約；model swap 邊界稀釋
- **防禦武器**：§B3 永恆禁區 / ADR-0028 AI Employee Charter / ADR-0047 Forbidden 集中 + 自動 guardrail test
- **現況**：黃燈 — Charter 已建但 Forbidden 散落 ADR / SKILL / safety_gate，Eval set 200 題未到位
- **合夥人投票**：× 認為過慮
- **會議備註**：
  > 每一家活做的廠商要把合約附上 每一家都不一樣
  > llm 是透過 文件去檢索 不是寫在 prompt 內

### F4 — 合規崩潰死 — 一次 PII 外洩或合約 4.4 不達標，甲方終止合約

- **情境**：甲方季度稽核發現負面情緒識別率只有 78%（合約要求 90%）；同時師傅 App 把客戶地址記在本地未加密，被資安公司紅隊測出。甲方依合約第 9 條終止。三個月後現金流斷裂。
- **死因關鍵**：合規視為「附加項」而非「主動脈」
- **防禦武器**：§B3 + §B4 金融完整性 + §B5 證據可稽核 + ADR-0050/0051 PII retention + Family Reviewer 角色
- **現況**：紅燈 — 合約 4.4(a) 90% 情緒識別 UAT 未通過、4.4(d) 家族覆核機制未建、PII retention engine 未建
- **合夥人投票**：× 認為過慮
- **會議備註**：
  > 現在採用 附加項

### F5 — 規模困境死 — 第二個甲方 / 海外擴張時要重寫 60% 程式

- **死因關鍵**：Tenant / Brand / Locale 沒在 day 1 設計進主檔

- **現況**：黃燈 — ADR-0030 已啟蒙但 locale 完全沒設計；Contract Template ADR 仍是草稿
- **合夥人投票**：✓ 認同是大風險
- **會議備註**：
  > 需要規劃第二個甲方 所有基本資料的 相關資料 需要獨立出來

### F6 — 人才流失死 — 資深技師 / Domain expert 離職，隱性知識消失

- **情境**：2027 年。資深技師 A 跳槽到競品，帶走他腦中對 50+ 品牌的 troubleshooting know-how。我們系統內的 SOP 螺旋只跑了一年，知識資產化 KPI 沒達標。新進客服上線後準確率掉到 65%，回到創業初期水準。
- **死因關鍵**：SOP 螺旋未閉環，知識仍依賴個人腦中
- **防禦武器**：§D2 知識護城河 + §E2 知識螺旋 S6 Human Correction + ADR-0038 SOP 雙審入庫
- **現況**：紅燈 — SOP 螺旋未閉環，Domain expert 角色 / 留任機制未定義
- **合夥人投票**：✓ 認同是大風險

## 二、GAP 跨團隊決策

### ADR-0031 — AI 能不能自己幫客人開工單？

- **痛點**：目前每張工單都要客服人工確認，每筆多花 5-10 秒。但 AI 自己開單一旦地址 / 品牌錯了，截圖傳開信任崩盤。
- **拍板**：**✓ 推薦做法** — AI 先草擬，客服按一下確認
  - 內容：AI 把資料填好，客服只需按 1-click 確認。HITL 邊界鎖死。
  - 代價：客服每筆多 5-10 秒；UI 多一個確認按鈕

### ADR-0032 — 客人不給地址，工單還能不能開？

- **痛點**：地址不全會讓師傅空跑、客戶不滿。但全部強制收齊，50% 客人會在地址這關放棄。
- **拍板**：**替代方案 A** — 工單可以先開，地址後補
  - 內容：派工後再追，師傅自己 confirm。
  - 代價：派錯地址 / 師傅空跑 / 對帳混亂
- **會議備註**：
  > 工單可以接受暫時不填寫住址 師父假如確認有這個客戶的地址它知道 它可以直接skip, 但是工單結案以前要把住址回填 

### ADR-0033 — 問題卡資料不齊全，要不要硬派工？

- **痛點**：問題卡分數 0~1，越完整越高。低分硬派工 → 師傅到場才知道狀況不對。
- **拍板**：**✓ 推薦做法** — 0.85 分是門檻，低於需客服 override
  - 內容：低分案件不自動派，走 Exception module 客服判斷。
  - 代價：低分案件多 1 步人工

### ADR-0034 — 什麼算「急件」？

- **痛點**：目前 urgent 標籤散在 PC / SLA / AI 轉真人多處，定義不一致。
- **拍板**：**✓ 推薦做法** — 4 類具名：被鎖門外 / 門內受困 / 安全風險 / 怒客高風險
  - 內容：進 Domain Event Catalog，新類型走 ADR review。
  - 代價：每年校準 1 次

### ADR-0035 — 保固期內 / 建案案件 AI 能報價嗎？

- **痛點**：AI 一旦給保固個案 final quote，等於替公司做了法律承諾，一次失準就法務糾紛。
- **拍板**：**✓ 推薦做法** — AI 可給範圍（如 NTD 1500-3500），永禁 final quote
  - 內容：教育性 range OK，個案承諾禁止。Guardrail 自動偵測 + regen。
  - 代價：客戶等真人多 5-15 分鐘

### ADR-0036 — 同一個對話可以有幾張問題卡？

- **痛點**：今天問鎖 A、明天問鎖 B、後天追 A 的進度。一個對話多張 PC 後續對帳會亂。
- **拍板**：**✓ 推薦做法** — 同一 active 問題一張卡；新症狀 / 新設備可另開
  - 內容：規則簡單覆蓋 90%，多設備家庭仍能服務。
  - 代價：新症狀判定要先問客戶

### ADR-0037 — AI 遠端解決完，對話什麼時候關？

- **痛點**：永遠不關，DB 累積殭屍對話；客服主動結案，工作量爆量。
- **拍板**：**✓ 推薦做法** — 客戶點「已解決」or 48h 沒回應自動關
  - 內容：留 audit 標 auto-closed；7 天內客人有訊息可 reopen。
  - 代價：對話 state machine 多一個狀態

### ADR-0038 — AI 寫的 SOP 草稿誰來審？

- **痛點**：全自動入庫會污染知識庫；全部雙審，知識更新慢一半。
- **拍板**：**✓ 推薦做法** — 高風險（報價/退款/法律）雙審，FAQ 類單審
  - 內容：客服主管 + Domain expert 共審高風險 SOP；FAQ 由 Knowledge Owner 單審。
  - 代價：雙審高風險多 1 工作日

## 三、P0 業務規則

### ADR-0039 — 客人取消怎麼收費？

- **痛點**：目前未決，導致 AR 模組阻擋 + 客服面對爭議沒一致說法。
- **拍板**：**✓ 推薦做法** — 5 階段：報價未確認免費 → 已派工未出發收檢測費 → 出發後收車馬費 → 到場後車馬+檢測 → 已施工按比例
  - 內容：金額 configurable per brand。Backend 自動推算階段。
  - 代價：5 階段要 UI 清楚呈現
- **會議備註**：
  > 已派工未出發 不收費
  > 出發後沒到場 收車馬費
  > 到場後無法施工 收 檢測費和車馬費
  > 以上是系統自動判定
  > 但是克服可以覆寫
  > 

### ADR-0040 — 退款誰能批准？

- **痛點**：小額退款卡主管 / 大額退款被客服輕易放行，都是合規風險。
- **拍板**：**替代方案 A** — 依責任歸屬分層（品牌 / 平台 / 師傅）
  - 內容：看誰負責決定誰批。
  - 代價：矩陣維度 ↑ 2x（5×3=15 種組合）

### ADR-0041 — 車馬費怎麼分？

- **痛點**：全給師傅平台沒收入，全平台收師傅生態崩潰。
- **拍板**：**✓ 推薦做法** — 80% 師傅 / 20% 平台（同區 500 / 跨區 800 / 遠距 1200）
  - 內容：進 Dispatcher Commission ledger。建商 / 品牌合約可 override。
  - 代價：平台機會成本 -20%

### ADR-0042 — 誰能看什麼資料？（RBAC 矩陣）

- **痛點**：18 個角色（客服 / 派工 / 師傅 / 會計 / 主管 / IT / 品牌 / 經銷 / 建商 / 家族覆核員 ...）的精確權限矩陣未閉環。
- **拍板**：**✓ 推薦做法** — 4 層原則固化 + 具體欄位 configurable
  - 內容：顧客 / 營運 / 財務 / 治理 4 層永遠不變；具體可看 / 可改 / 可核准透過 RBAC engine 動態設定。
  - 代價：RBAC engine 開發成本中等
- **會議備註**：
  > 設定要透過後台可以轉換管理

### ADR-0043 — 品牌 / 建商合約怎麼掛？

- **痛點**：30 個品牌每個都不同合約：SLA / 月結 / 保固 / 可見性。寫死會崩，靠外部 CRM 主權外移。
- **拍板**：**✓ 推薦做法** — 合約模板物件（Contract Template）+ tenant scope
  - 內容：每個 partner 對應一個合約 instance，內含 SLA / 月結 / 保固 / 責任分配。新 partner 走 configuration 零 code change。
  - 代價：Contract Template schema 要謹慎設計

### ADR-0044 — 保固從哪天開始算？

- **痛點**：B2C 是購買日；建商案件要點交日；專案案件要啟用日。寫死「購買日」會傷建案。
- **拍板**：**✓ 推薦做法** — Device 加 warranty_start_mode 欄位（5 模式可選）
  - 內容：purchase_date / handover_date / activation_date / contract_date / manual_override 五選一。
  - 代價：Device schema 多 2 欄 + migration

### ADR-0045 — 師傅多久要接單？

- **痛點**：全平台統一不能調整 VIP 品牌會跑；per region 浮動 KPI 不可比。
- **拍板**：**✓ 推薦做法** — 一般 10 分鐘 / 急件 5 分鐘 + 合約可 override
  - 內容：Per brand contract 可自訂（VIP 客戶 5 min）。
  - 代價：SLA monitor 要支援 per-contract override

## 四、AI 邊界 + 治理

### ADR-0046 — 改價格 / 改權限 / 改規則的流程

- **痛點**：目前散落各處（後台 UI / 直接改 DB / 口頭）。Phase II Finance 一啟動就會「上週改的價格是誰核准的？」
- **拍板**：**✓ 推薦做法** — ChangeRequest 物件化（申請 → 核准 → 生效日 → audit）
  - 內容：所有政策 / 價格 / 權限 / SLA / 模板變更走此 workflow。緊急可走 emergency track。
  - 代價：ChangeRequest UI 要建

### ADR-0047 — AI 哪些事絕對不能做？（Forbidden 清單）

- **痛點**：「AI 可以做 X 嗎？」散落 ADR / PRD / SKILL.md / safety_gate code。每次新 skill 都要重新對齊。
- **拍板**：**✓ 推薦做法** — Forbidden 集中入 ADR-0028 charter + 自動測試把關
  - 內容：Final quote / 退款核准 / 保固判定 / 法律承諾 / 跨租戶 / 直接呼叫 convert_to_wo 等全列入。每次部署跑 200 題 Eval pass<95% block。
  - 代價：Eval pipeline 要建 + 每次部署 +5-10 min

### ADR-0048 — AI 什麼時候該轉真人？

- **痛點**：目前 prompt 引導 + LLM 自判，模糊。該轉沒轉 → 邊界漂移；硬轉 → AI 投資沒效益。
- **拍板**：**✓ 推薦做法** — 7 條硬規則：急件 / 怒客 / 高金額 / 保固不明 / 退款 / 法律安全 / 3 次失敗
  - 內容：任一觸發強制 transfer_to_human，沒有 LLM judgment 餘地。Eval 200+ 題回歸。
  - 代價：轉真人率 +5-10%

### ADR-0049 — 現場加價怎麼防爭議？

- **痛點**：師傅到場發現要加價，目前流程不一致（口頭 / 拍照 / 事後補）。爭議發生無法追溯。
- **拍板**：**✓ 推薦做法** — 三件套：客戶簽名 + Evidence 照片 + audit log
  - 內容：缺一不可。金額分層：≤500 師傅可確認 / 501-2000 客服 LINE 確認 / >2000 主管核可 + 三方在線。
  - 代價：加價流程 +2 分鐘

### ADR-0050 — Evidence（照片 / 簽名 / 發票）誰能看？

- **痛點**：Evidence 涉及 PII + 商業機密。客戶不該看成本拆分；師傅不該看其他人案件；品牌商不該看競品。
- **拍板**：**✓ 推薦做法** — 角色 × 案件生命週期 × 屬性過濾矩陣
  - 內容：客戶看自家工單施工照 + 簽名（90 天）；師傅看自己負責的（30 天）；會計看財務 Evidence（永久）；稽核唯讀全看。
  - 代價：可見性 engine 複雜度 +20%

### ADR-0051 — Evidence 要保存多久？

- **痛點**：太短法律糾紛沒證據；太長 GDPR / 個資法違反 + 儲存成本爆炸。
- **拍板**：**✓ 推薦做法** — 1 年預設 / RMA+3 年 / 法律相關永久 / 客戶刪除 7 天執行
  - 內容：依案件類型自動分層；客戶 GDPR forget 7 天內執行。
  - 代價：Retention 判定要在 Evidence 建立時設定 mode

### ADR-0052 — 材料 / 零件由誰買誰收？

- **痛點**：師傅墊付材料費對帳難、品牌商保固件不算成本、平台採購批量料用於多 case。
- **拍板**：**✓ 推薦做法** — Material 加 owner 欄位：platform / brand / locksmith 三選一
  - 內容：月結時依 owner 自動分流到三個 ledger。
  - 代價：Material schema 多 1 欄 + 師傅 App 加發票上傳

### ADR-0053 — 什麼零件要記序號（Serial）？

- **痛點**：全部強制 serial 現場流程爆炸；全部選填保固爭議無法追。
- **拍板**：**✓ 推薦做法** — 主鎖 + 高價零件（>1000）強制，低價選填
